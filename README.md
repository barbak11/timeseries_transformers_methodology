# Timeseries Transformers Methodology

**Разработка комплексного подхода применения трансформеров для прогнозирования временных рядов сложной природы.**

Суть проекта — проверить, как ведут себя разные классы моделей (классические методы, бустинг, нейросети и трансформеры) в задачах с различной структурой данных. Основной фокус сделан не на одной конкретной архитектуре, а на сравнении подходов в разных условиях и на формировании практических рекомендаций по выбору модели.

Эксперименты охватывают несколько типов временных рядов: разреженные данные, короткая история, холодный старт и группы взаимосвязанных рядов. Для каждого сценария оценивается, какой класс моделей лучше справляется с задачей и когда использование более сложных Transformer-архитектур действительно оправдано.

Практическая цель работы:

- понять, дает ли трансформер устойчивый прирост качества;
- определить, в каких постановках достаточно бустинга или классических моделей;
- оценить, как структура данных влияет на итоговый результат.

Сравнение проводится не только по точности прогноза, но и по вычислительным затратам, чтобы выводы были применимы в реальных условиях.

---

## Обзор

Проект ориентирован на прикладное сравнение моделей для разных типов данных:

- классические методы (`ETS`, `SARIMAX`);
- бустинг (`XGBoost`);
- нейросетевые модели (`LSTM`, `TFT`, `PatchTST`, MV-Transformer);
- специализированные сценарии:
  - `sparse` (разреженные продажи),
  - `short` (короткая история),
  - `coldstart` (новые/недостаточно длинные ряды),
  - `correlated` (кластер коррелированных рядов).

### Две решаемые задачи (traffic / disease)

Во всех спецификах эксперименты организованы вокруг **двух задач**:

- **traffic**: прогноз целевой переменной по ресторанным временным рядам.
- **disease**: прогноз по эпидемиологическим данным.

## Ключевые возможности

1. **Сравнение нескольких классов моделей** в едином кодовом стиле и с сохранением метрик по экспериментам.
2. **Поддержка разных постановок задачи** отдельными скриптами (`sparse`, `short`, `coldstart`, `correlated`).
3. **Единый конфиг** в `config.py` для общих путей и базовых гиперпараметров.
4. **Общий модуль повторно используемой логики** в `common/short_shared.py` для пересекающихся функций.
5. **Сохранение результатов** в `artifacts/<spec>/...` (модели, предсказания, метрики).

## Структура проекта

```text
timeseries_transformers_methodology/
├─ config.py                  # единая конфигурация путей и общих параметров
├─ sparse.py                  # единый пайплайн для sparse (включая PatchTST)
├─ short.py                   # эксперименты для коротких рядов
├─ coldstart.py               # эксперименты cold-start
├─ correlated.py              # эксперименты для коррелированных рядов
├─ common/
│  └─ short_shared.py         # общий код для short/coldstart
├─ functions/
│  └─ metrics.py              # общие метрики (rmse/wape/mae и т.д.)
├─ requirements.txt
├─ LICENSE
└─ artifacts/                 # результаты экспериментов (в git, кроме моделей)
```

## Установка

Требуется Python 3.10+ (рекомендуется 3.11).

1. Клонируйте репозиторий:

```bash
git clone <URL_ВАШЕГО_РЕПОЗИТОРИЯ>
cd timeseries_transformers_methodology
```

2. Создайте и активируйте виртуальное окружение:

```bash
python -m venv .venv
```

Windows (PowerShell):

```bash
.venv\Scripts\Activate.ps1
```

Linux/macOS:

```bash
source .venv/bin/activate
```

3. Установите зависимости:

```bash
pip install -r requirements.txt
```

## Быстрый старт

Запускать можно каждый сценарий отдельно:

```bash
python sparse.py
python short.py
python coldstart.py
python correlated.py
```

`PatchTST` для sparse и disease-сценариев реализован в `sparse.py` и запускается через флаги `do_patchtst`.

После запуска результаты сохраняются в `artifacts/<spec_key>/`.

## Данные и артефакты

По умолчанию пути к данным заданы в `config.py`:

- `data/` — основные parquet-таблицы трафика и sparse-данные;
- `Zab/processed/` — подготовленные наборы для disease/PCR-постановок.

Артефакты экспериментов сохраняются в:

- `artifacts/<spec_key>/models`
- `artifacts/<spec_key>/predictions`
- `artifacts/<spec_key>/metrics`

---

## Конфигурация

Базовые общие параметры и пути вынесены в `config.py`:

- пути к датасетам;
- корень артефактов;
- общие гиперпараметры (`RANDOM_SEED`, `TEST_HORIZON_YEARS`, параметры `XGBoost`/`LSTM`);
- общие признаки и целевая переменная.

Специфичные параметры каждого сценария остаются в соответствующих скриптах.

---

## Рекомендации по воспроизводимости

- фиксируйте `RANDOM_SEED`;
- используйте одинаковую версию Python и `requirements.txt`;
- не смешивайте артефакты разных запусков в одной папке без маркировки;
- храните большие данные и модели вне git (это уже настроено через `.gitignore`).
- артефакты экспериментов можно хранить в git, но папки `models/` внутри `artifacts/` игнорируются (чтобы не раздувать репозиторий).
